## Plan Generated: Bugfix - Consistent Match Summary Between Logs and Excel Output

### 目标
- 解决“日志匹配统计”与“写入 Excel 的成功/失败计数”不一致的问题，确保两者来自同一个数据源，且在处理完成后输出一致的汇总信息。

### 背景与现状
- 日志显示"共处理 10 条，匹配成功 0 条，失败 0 条"，但写入 Excel 的成功/失败条数与日志不一致，存在统计口径不一致和潜在的竞态条件。
- 可能原因包括：并发写入、独立的计数变量、异常路径未正确记入、以及日志与写入 Excel 的数据源不同步等。

### 假设与风险
- 假设 A：有一个统一的结果数据结构记录每条任务的状态（成功/失败/跳过等）。
- 假设 B：Excel 写入在处理完成后统一执行，且使用同一结果集合生成汇总。
- 风险：若原有代码中分支多、错误处理零散，改动可能引入回归；需要通过单元/端到端测试来验证。

### 诊断计划 (Phase 1)
- 复现并对比：用同一数据集在日志与 Excel 中输出结果，确保能触发统计不一致的分支。
- 收集证据：对每条记录在内存中的状态、日志输出、Excel 行的内容进行逐条对照。
- 确认数据源：确认日志计数与 Excel 汇总的真实数据来源是否相同。

### 实施要点 (Phase 2 – 变更点，决策完成)
- 引入统一的结果记录器：ResultsRecorder，提供方法 add_result(id, status) 与 finalize_and_write()，确保所有条目在写入 Excel 之前都被记录。
- 日志与汇总规则统一：日志的“成功/失败”计数改为从 ResultsRecorder 的最终结果中计算得到的派生值，而非独立变量。
- 处理异常路径：若单条处理抛出异常，确保该条目被记录为失败，并写入 Excel，避免丢失条目。
- 并发保护：若使用多线程/异步执行，确保对结果集合的写入与计数具备线程安全性（加锁或使用线程安全结构）。
- 最终汇总输出：在完成阶段只输出来自结果集合的汇总，避免双重统计。
- 回滚与向前兼容：若新实现不可用，保留老的日志路径作为兜底，但不再输出误导性汇总。

### 验证用例 (Phase 3)
- 单元测试：不同状态组合的 ResultsRecorder 行为（全部成功、全部失败、混合、边界值）。
- 端到端测试：模拟 10 条任务的处理流程，确保日志显示的统计与 Excel 文件中的统计完全一致。
- 负载测试：在高并发下确保统计与写入的一致性，不产生竞态条件导致的错位计数。

### 验收准则
- 日志中的汇总应等于结果集合中的计数总和，且等于 Excel 导出的汇总。
- 任意条目若被处理失败，Excel 与日志都应正确记录失败状态。
- 不再出现日志计数与 Excel 计数不一致的情况。

### 变更范围与影响
- 影响：涉及结果记录、日志输出、Excel 写入逻辑、以及异常处理分支。
- 风险控制：新增的 ResultsRecorder 需经过单元和端到端测试，确保与现有代码接口兼容。

### 路线图与里程碑
- Wave 1：添加 ResultsRecorder，替换独立计数变量为基于结果集合的统计。
- Wave 2：对并发写入做保护，确保线程安全。
- Wave 3：实现 finalize/write 的单点出口，统一日志汇总。
- Wave 4：编写并执行端到端测试用例，验证日志与 Excel 的一致性。
